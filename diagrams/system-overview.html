System overview

How Powershell REST JobRunner works

A producer hands CRUD jobs to a shared store and a queue; a pool of runspace workers drains the queue in parallel, running each job through a five-step request pipeline against the REST API, with retry and structured outcomes. Below: the runtime flow in motion, the job lifecycle, the pipeline, and how every piece built here fits together.

01

Runtime data flow

jobs moving through the engine, concurrently
a job (CRUD request) data flow retry / exceptional path

The invariant that makes it safe: only job ids travel through the queue, and each id is taken by exactly one worker — so although many jobs run at once, no single job record is ever written by two threads. Every status change is funnelled through one choke point that also updates the shared store.

02

Job lifecycle

the states a single job moves through

Terminal states are Succeeded, Failed, DeadLettered and Cancelled. The split between Failed and DeadLettered is what kind of error stopped the job: a non-retryable one (like validation) fails immediately, while a retryable one that runs out of attempts is dead-lettered.

Retry happens here

A transient failure at SendRequest (timeouts, 408/429/5xx) is retried up to MaxAttempts with capped exponential backoff.

Fail fast

A validation error at Validate is non-retryable — the job goes straight to Failed without contacting the API.

Skip on failure

When a step throws, every later step is marked Skipped, so the report shows exactly where the job stopped.

Each attempt is clean

Before retrying, all five steps reset to Pending, so an attempt always starts from a known state.

04

How the pieces fit together

the artifacts built around one API

One contract, three companions: the spec documents the API, the reference page renders that spec, the Mockoon mock implements it as a runnable server, and RestJobRunner is a client that drives it with concurrent CRUD jobs.
